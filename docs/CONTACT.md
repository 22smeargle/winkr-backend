## Reporting Issues

### Issue Reporting Guidelines

#### Before Creating an Issue
1. **Search Existing Issues**: Check if issue already exists
2. **Check Documentation**: Review documentation for solutions
3. **Test Latest Version**: Ensure issue exists in latest version
4. **Reproduce Issue**: Provide clear reproduction steps
5. **Check Environment**: Verify issue is not environment-specific

#### Issue Categories
- **Bug Report**: Unexpected behavior or errors
- **Feature Request**: New functionality suggestions
- **Documentation**: Documentation issues or improvements
- **Performance**: Performance-related issues
- **Security**: Security vulnerabilities (see Security section)
- **Question**: General questions (use Discussions instead)

#### Issue Template
```markdown
## Issue Type
- [ ] Bug
- [ ] Feature Request
- [ ] Documentation
- [ ] Performance
- [ ] Other

## Description
Clear description of the issue or feature request

## Steps to Reproduce
1. Step 1
2. Step 2
3. Step 3

## Expected Behavior
What should happen

## Actual Behavior
What actually happens

## Environment
- OS: [e.g., macOS, Linux, Windows]
- Go Version: [e.g., 1.21.0]
- Database: [e.g., PostgreSQL 14]
- Redis: [e.g., Redis 6.2]

## Additional Context
Any additional information, screenshots, or logs
```

### Security Reporting

#### Vulnerability Disclosure
- **Private Reporting**: Email security issues to [plus4822@icloud.com](mailto:plus4822@icloud.com)
- **Response Time**: Acknowledgment within 24 hours
- **Resolution Timeline**: Patches within 7-14 days
- **Disclosure**: Coordinated disclosure with reporter

#### Security Policy
- **Scope**: Security issues in Winkr Backend code
- **Out of Scope**: Third-party dependencies, user configurations
- **Reward**: Recognition and potential bounty for critical issues
- **Process**: Follow responsible disclosure guidelines

#### Security Categories
- **Critical**: Remote code execution, data breaches
